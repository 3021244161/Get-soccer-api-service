# 足球数据抓取缓存 API 接口文档

## 1. 文档说明

本文档描述足球抓取缓存 API 的使用方式、鉴权规则、接口清单、请求参数、响应格式、错误码和字段定义。

接口覆盖以下四个模块：

- `jczq`：竞彩足球
- `bjdc`：北京单场
- `zq14`：足球14场
- `sfgg`：胜负过关

服务定位：

- 提供 REST + JSON 风格的只读查询接口
- 提供后台刷新与刷新状态查询接口
- 通过缓存承接高并发读取
- 提供 API Key 鉴权能力

数据更新方式：

- 服务端按固定周期执行后台刷新
- 查询接口默认只读取当前激活版本缓存
- 刷新失败时继续保留上一版成功缓存

## 2. 基础信息

### 2.1 Base URL

本地示例：

```text
http://127.0.0.1:8000
```

公网隧道示例：

```text
https://juniper-covenant-slaw.ngrok-free.dev
```

### 2.2 数据格式

- 请求体：`application/json`
- 响应体：`application/json; charset=utf-8`

### 2.3 鉴权方式

除健康检查接口外，所有业务接口均要求通过请求头传递 API Key：

```http
X-API-Key: your_api_key
```

如果通过 `ngrok` 免费域名对外访问，为跳过 `ngrok` 浏览器警告页，调用方还需要额外携带：

```http
ngrok-skip-browser-warning: true
```

说明：

- 本地直接调用时不需要该请求头
- 通过 `ngrok` 公网域名调用 API 时建议统一添加该请求头
- 浏览器地址栏直接访问时通常无法自定义该请求头，更适合使用 Postman、curl、脚本或前端代理调用

### 2.4 时间与字符编码

- 所有响应使用 UTF-8 编码
- 时间字段使用字符串返回
- 不同字段的具体时间格式以实际字段定义为准

### 2.5 字段类型约定

比赛详情对象中的所有字段统一按字符串返回：

- 数值按字符串返回
- 布尔含义字段按字符串返回
- 列表和对象结构按 JSON 字符串返回
- 取不到的字段返回空字符串 `""`

这是当前接口返回约定，调用方不应假设这些字段会以数字、布尔或嵌套对象形式返回。

## 3. 模块编码

| 模块编码 | 中文名称 | 说明 |
| --- | --- | --- |
| `jczq` | 竞彩足球 | 足球竞彩模块 |
| `bjdc` | 北京单场 | 仅列表玩法取 `胜平负` |
| `zq14` | 足球14场 | 当源站无有效比赛时允许返回空列表 |
| `sfgg` | 胜负过关 | 抓取深度与其他足球模块一致 |

## 4. 鉴权规则

### 4.1 需要鉴权的接口

以下接口要求传 `X-API-Key`：

- `GET /api/v1/meta`
- `GET /api/v1/matches`
- `GET /api/v1/modules/{module_code}`
- `GET /api/v1/modules/{module_code}/matches`
- `GET /api/v1/modules/{module_code}/matches/{match_id2}`
- `POST /api/v1/admin/refresh`
- `GET /api/v1/admin/refresh/status`

### 4.2 无需鉴权的接口

- `GET /health`

### 4.3 鉴权失败语义

- 未传 `X-API-Key`：返回 `401`
- `X-API-Key` 无效：返回 `403`

## 6. meta 字段说明

`meta` 为接口响应外层元数据对象，常见字段如下：

| 字段名 | 含义 |
| --- | --- |
| `version` | 当前命中的缓存版本号 |
| `generated_at` | 当前返回数据对应的生成时间 |
| `cache_status` | 当前缓存状态 |
| `refresh_interval_seconds` | 后台刷新周期，单位秒 |
| `fast_refresh_interval_seconds` | 快刷新周期，单位秒 |
| `slow_refresh_interval_seconds` | 慢刷新周期，单位秒 |
| `request_id` | 请求唯一标识 |
| `module_code` | 当前返回所属模块，部分接口提供 |
| `last_refresh_mode` | 最近一次刷新模式 |
| `last_fast_success_refresh_at` | 最近一次快刷新成功时间 |
| `last_slow_success_refresh_at` | 最近一次慢刷新成功时间 |

`cache_status` 常见取值：

| 值 | 含义 |
| --- | --- |
| `fresh` | 当前数据为最新成功刷新版本 |
| `stale` | 当前数据为旧版本缓存 |
| `empty` | 当前还没有成功缓存版本 |
| `degraded` | 服务处于降级状态 |

## 7. 状态码

| 状态码 | 含义 |
| --- | --- |

## 10. 缓存规则

- 快刷新周期：`720` 秒
- 慢刷新周期：`7200` 秒
- 查询接口始终读取当前 `active_version`
- 刷新失败时继续返回上一版成功缓存
- 默认保留最近多个成功版本

双频刷新说明：

- `fast`：刷新赛事赔率、欧指、亚指等高频字段
- `slow`：刷新战绩、综合实力、阵容等低频字段
- `full`：启动阶段或人工触发时执行完整刷新
- 对外查询接口结构保持不变，调用方始终读取聚合后的完整比赛对象

缓存版本说明：

- 每轮成功刷新生成新的 `version`
- 版本切换完成后，查询接口开始命中新版本
- 查询接口返回的 `meta.version` 即当前命中的版本号

## 11. debug 字段裁剪规则

当请求参数 `debug=false` 时，接口会裁剪大字段，减小响应体积。

建议被裁剪字段如下：

- `history_dom_text`
- `strength_detail_dom_text`
- `lineup_dom_text`
- `europe_dom_text`
- `asia_dom_text`
- `history_api_raw`
- `base_api_raw`
- `lineup_api_raw`
- `europe_list_api_raw`
- `europe_stats_api_raw`
- `asia_list_api_raw`
- `asia_stats_api_raw`
- `request_log`

适用场景：

- 高并发查询
- 列表页展示
- 只做概要分析，不看调试原文

## 13. 轮询与校验建议

如果调用方需要持续读取并检查字段是否遗漏，建议按如下流程：

1. 调用 `POST /api/v1/admin/refresh` 触发刷新
2. 轮询 `GET /api/v1/admin/refresh/status`，直到 `status=idle` 且 `active_version` 非空
3. 调用 `GET /api/v1/modules/{module_code}/matches` 获取比赛列表
4. 选取一场比赛，调用 `GET /api/v1/modules/{module_code}/matches/{match_id2}`
5. 间隔几秒后重复读取同一场比赛
6. 比较两次读取的字段总数、空字段数量和字段名集合

建议重点检查：

- 第二次读取是否缺字段
- 第二次读取是否新增错误信息
- 关键字段是否从有值变为空

- 隧道停止、电脑重启或重新建立隧道后，域名可能变化
- 给客户试用时，建议同时提供 `X-API-Key` 与 `ngrok-skip-browser-warning` 请求头要求

## 15.5 API Key 管理建议

建议采用以下方式管理：

- 为每个调用方分配独立 `API Key`
- 开发、测试、客户试用、正式环境使用不同 key
- `API_KEYS` 环境变量支持逗号分隔多个 key
- 替换或撤销某个 key 后，重启服务使配置生效
- 不要把正式 key 写死在前端页面或公开仓库中

示例：

```text
API_KEYS=dev-key,trial-key-001,prod-key-001
```
